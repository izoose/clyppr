# Clipper M0 — Engine Viability Spikes: Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Prove the two hard, load-bearing parts of a custom C# capture engine work on *this* machine before building the app around them: (S1) live screen capture encoded to H.264 via NVENC, and (S2) per-application audio capture producing physically separated tracks.

**Architecture:** Two small .NET 8 console programs in one solution. S1 captures the primary monitor with Windows Graphics Capture (WGC via Vortice.Windows), reads each frame back to CPU, and pipes raw BGRA to the already-installed `ffmpeg` (`h264_nvenc`) which encodes + muxes a 10 s MP4. S2 uses WASAPI process-loopback (`ActivateAudioInterfaceAsync` + `AUDIOCLIENT_ACTIVATION_PARAMS`) to capture one target process (and its tree) *included* and, in parallel, *everything except* that process, writing two WAV files. Verification is automated with `ffprobe` plus a manual playback check.

**Tech Stack:** C# / .NET 8 (`net8.0-windows10.0.20348.0`), Vortice.Windows (D3D11/DXGI + WGC interop), Windows SDK WinRT projections (Windows.Graphics.Capture), WASAPI process-loopback via P/Invoke, bundled system `ffmpeg`/`ffprobe` (v8.1, NVENC).

## Global Constraints

- Target framework floor: `net8.0-windows10.0.20348.0` (build 20348 is the minimum for WASAPI process-loopback; this machine is 26200).
- Machine reality this plan is written against: RTX 3060 (NVENC H.264/HEVC), ffmpeg 8.1 full build on PATH with `h264_nvenc`, ffprobe on PATH, .NET 8.0.422 SDK.
- These are **spikes**: the success bar is "the primitive demonstrably works and produces a correct file," not production quality. Do not add config, UI, or abstraction. Delete-friendly code is fine.
- Each task ends by committing. `git init` happens in Task 0.
- Verification prefers `ffprobe -v error -print_format json -show_streams -show_format <file>` assertions over manual inspection, but every media spike also gets one human "play it and confirm" check.

## Gate (the point of M0)

After both tasks: **both spikes pass → the custom C# engine is viable; proceed to write the M1 plan.** If S2 (audio) is a wall, fall back to OBS-driven audio or reconsider the engine choice. If S1 (video) is a wall, switch the encode path (Media Foundation SinkWriter zero-copy, or Desktop Duplication capture) and re-spike.

## Task 1 — Spike S1: WGC screen capture → NVENC → 10 s MP4

**Goal:** Capture the primary monitor for ~10 seconds and produce `out/spike.mp4` encoded by `h264_nvenc`.

**Files:**
- Create: `spikes/S1.VideoCapture/WgcCapture.cs` (WGC device + frame pool + FrameArrived → staging readback → raw BGRA callback)
- Modify: `spikes/S1.VideoCapture/Program.cs` (wire capture → ffmpeg stdin, run 10 s, stop)

**Interfaces:**
- Produces (`WgcCapture`):
  - `WgcCapture(int monitorIndex = 0)` — picks the primary output.
  - `int Width { get; }`, `int Height { get; }` — capture surface size (set after `Start`).
  - `event Action<byte[]> FrameBgra` — fires per frame with a tightly-packed BGRA buffer of `Width*Height*4` bytes (row-stride removed).
  - `void Start()` / `void Stop()`.

**Approach notes (read before coding — this is the fiddly part):**
- WGC needs a `GraphicsCaptureItem`. There is no pure-managed constructor; you obtain it through the COM interop interface `IGraphicsCaptureItemInterop` (`CreateForMonitor(HMONITOR)`), which is exposed by the `GraphicsCaptureItem` activation factory. Get the HMONITOR from `MonitorFromWindow`/`EnumDisplayMonitors` (primary = `MonitorFromPoint({0,0}, MONITOR_DEFAULTTOPRIMARY)`).
- Create a Vortice D3D11 device (`D3D11.D3D11CreateDevice`, BgraSupport flag). Wrap it as an `IDirect3DDevice` for WGC via `CreateDirect3D11DeviceFromDXGIDevice` (interop from `Windows.Graphics.DirectX.Direct3D11.Interop`).
- `Direct3D11CaptureFramePool.Create(device, DirectXPixelFormat.B8G8R8A8UIntNormalized, 2, size)` → `CreateCaptureSession(item)` → `session.StartCapture()`; handle `FrameArrived`.
- Per frame: `frame.Surface` → get `ID3D11Texture2D` (interop `IDirect3DDxgiInterfaceAccess.GetInterface`), `CopyResource` into a CPU-readable **staging** texture (`Usage.Staging`, `CpuAccessFlags.Read`), `Map`, copy row-by-row removing `RowPitch` padding → raise `FrameBgra`.
- This interop is well-trodden but exacting; if a specific interop cast fights you, that *is* the spike doing its job. Iterate here — do not abstract.

- [ ] **Step 1: Write `WgcCapture.cs`**

Implement the class per the Interfaces + Approach above. Core FrameArrived body (the load-bearing loop):
```csharp
private void OnFrameArrived(Direct3D11CaptureFramePool pool, object args)
{
    using var frame = pool.TryGetNextFrame();
    if (frame is null) return;

    using var srcTex = GetTexture(frame.Surface);           // interop unwrap to ID3D11Texture2D
    if (_staging is null) CreateStaging(srcTex.Description); // lazily size staging + set Width/Height
    _context.CopyResource(_staging, srcTex);

    var map = _context.Map(_staging, 0, MapMode.Read, Vortice.Direct3D11.MapFlags.None);
    try
    {
        var buffer = new byte[Width * Height * 4];
        unsafe
        {
            byte* src = (byte*)map.DataPointer;
            fixed (byte* dst = buffer)
                for (int y = 0; y < Height; y++)
                    Buffer.MemoryCopy(src + y * map.RowPitch, dst + y * Width * 4, Width * 4, Width * 4);
        }
        FrameBgra?.Invoke(buffer);
    }
    finally { _context.Unmap(_staging, 0); }
}
```

- [ ] **Step 2: Write `Program.cs` — pump frames into ffmpeg**

```csharp
var capture = new WgcCapture();
capture.Start();                       // sets Width/Height after first frame; spin-wait until known
while (capture.Width == 0) Thread.Sleep(10);

Directory.CreateDirectory("out");
var args = $"-y -f rawvideo -pixel_format bgra -video_size {capture.Width}x{capture.Height} " +
           $"-framerate 60 -i - -c:v h264_nvenc -preset p5 -pix_fmt yuv420p -t 10 out/spike.mp4";
using var proc = Process.Start(Ffmpeg.Start(args))!;
var stdin = proc.StandardInput.BaseStream;

capture.FrameBgra += buf => { try { stdin.Write(buf, 0, buf.Length); } catch { } };

Thread.Sleep(10_500);                  // ~10 s of frames
capture.Stop();
stdin.Close();                         // signals EOF; ffmpeg finalizes the mp4
proc.WaitForExit();
Console.WriteLine("done -> out/spike.mp4");
```
(ffmpeg drops frames if delivery is uneven; that's fine for the spike. `-t 10` caps duration.)

## Task 2 — Spike S2: WASAPI process-loopback → two separated audio tracks

**Goal:** Given a target process name (default `chrome` for easy testing — plays audio, easy to launch; swap to `Discord`/`RobloxPlayerBeta` when testing the real scenario), capture **10 s** into two WAVs simultaneously: `out/included.wav` (only the target process tree) and `out/excluded.wav` (everything *except* the target process tree). Play a sound in the target app during capture; `included.wav` should contain it and `excluded.wav` should not (and vice-versa for other apps).

**Files:**
- Create: `spikes/S2.AudioSplit/ProcessLoopbackNative.cs` (P/Invoke: `ActivateAudioInterfaceAsync`, activation params structs, completion handler, WASAPI COM interfaces: `IAudioClient`, `IAudioCaptureClient`)
- Create: `spikes/S2.AudioSplit/ProcessLoopbackCapture.cs` (managed wrapper: start one loopback capture in INCLUDE or EXCLUDE mode for a PID, expose a PCM callback)
- Create: `spikes/S2.AudioSplit/WavWriter.cs` (minimal 32-bit-float or 16-bit PCM WAV writer)
- Modify: `spikes/S2.AudioSplit/Program.cs` (resolve target PID, run two captures for 10 s, write two WAVs)

**Interfaces:**
- Produces (`ProcessLoopbackCapture`):
  - `enum LoopbackMode { IncludeTree, ExcludeTree }`
  - `ProcessLoopbackCapture(uint targetPid, LoopbackMode mode)`
  - `WaveFormat Format { get; }` — the negotiated mix format (expected 48 kHz, 2 ch, IEEE float).
  - `event Action<byte[]> DataAvailable` — raw PCM in `Format`.
  - `void Start()` / `void Stop()`.
- Produces (`WavWriter`): `WavWriter(string path, int sampleRate, int channels, int bitsPerSample, bool isFloat)`, `void Write(byte[] pcm)`, `IDisposable`.

**Approach notes (this is the novel, riskiest part):**
- Activation string: the device interface path constant `VIRTUAL_AUDIO_DEVICE_PROCESS_LOOPBACK` = `"VAD\\Process_Loopback"`. Pass to `ActivateAudioInterfaceAsync(path, IID_IAudioClient, &activationParams, completionHandler, &op)`.
- `activationParams` is a `PROPVARIANT` of type `VT_BLOB` wrapping `AUDIOCLIENT_ACTIVATION_PARAMS { ActivationType = ProcessLoopback(=1); ProcessLoopbackParams { TargetProcessId = pid; ProcessLoopbackMode = IncludeTargetProcessTree(0) | ExcludeTargetProcessTree(1) } }`.
- `ActivateAudioInterfaceAsync` is async: implement `IActivateAudioInterfaceCompletionHandler`, wait on an event, then `GetActivateResult` → `IAudioClient`.
- Initialize the client in **shared, loopback, event-driven** mode: `Initialize(AUDCLNT_SHAREMODE_SHARED, AUDCLNT_STREAMFLAGS_LOOPBACK | AUDCLNT_STREAMFLAGS_EVENTCALLBACK, hnsBufferDuration, 0, &format, null)` with a 48 kHz / 2 ch / 32-bit-float `WAVEFORMATEX`. Get `IAudioCaptureClient` via `GetService`. Set the event handle (`SetEventHandle`), `Start()`, then loop: wait on handle → `GetBuffer`/`ReleaseBuffer`, raising `DataAvailable` (emit silence when the `AUDCLNT_BUFFERFLAGS_SILENT` flag is set so both files stay time-aligned).
- **Reference implementation to mirror:** Microsoft's `ApplicationLoopback` Windows sample (the C++ one). Port its activation + capture loop to C#. Known-good C# ports exist; use one as a cross-check rather than deriving PROPVARIANT/COM marshalling blind. Treat exact marshalling as spike iteration — the *gate* is the two output WAVs, not code elegance.
